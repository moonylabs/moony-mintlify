<div style={{ textAlign: 'justify' }}>

<p style={{ marginBottom: '1.5rem' }}>
Moony's primary price discovery mechanism is an onchain bonding curve - a deterministic exponential function encoded within the Moony Reserve smart contract. Unlike traditional market-based price discovery that relies on order books or liquidity pools, the bonding curve calculates the price of MNY based solely on the current circulating supply. As tokens enter circulation through unlocking, the price rises along a mathematically defined trajectory. As tokens are redeemed and removed from circulation, the price falls along the same curve in reverse.
</p>

<p style={{ marginBottom: '1.5rem' }}>
The exponential model creates predictable incentive structures: gradual price appreciation during early adoption, accelerating value capture as the network matures, and a natural ceiling at maximum supply. Early participants benefit from lower entry prices while later participants contribute proportionally more capital to the reserve - capital that backs the redemption value of all circulating tokens. This mechanism applies symmetrically to both unlocking and redemption, enforced by immutable onchain code with no administrative override capability. All transaction outcomes are independently verifiable using the formulas documented below.
</p>

</div>

## Spot Price

<div style={{ textAlign: 'justify' }}>

<p style={{ marginBottom: '1.5rem' }}>
The spot price represents the instantaneous price per MNY token at any given circulating supply level - a continuous mathematical function with no bid-ask spread and no dependence on external liquidity. It is defined by an exponential equation with three immutable constants:
</p>

</div>

  marginBottom: '1.5rem'

  <div className="formula-box">
    $$
    P(S) = a \times b \times e^{c \times S}
    $$
  </div>

</p>

## Transaction Pricing

<div style={{ textAlign: 'justify' }}>

<p style={{ marginBottom: '1.5rem' }}>
The spot price tells you the current price per token, but you cannot simply multiply it by the quantity you want - because each token you buy slightly increases the price of the next one. Instead, the total cost is computed by integrating the spot price function across the full range of the transaction. The formulas below handle this automatically: two for buying (cost and tokens received), two for selling (value and tokens required).
</p>

</div>

### Cost to Buy Tokens

<div style={{ textAlign: 'justify' }}>

<p style={{ marginBottom: '1.5rem' }}>
Given a token quantity to acquire, this formula computes the total USDF cost - the definite integral from current supply to new supply:
</p>

</div>

<div className="principle-card formula-card" style={{
  padding: '1.5rem',
  marginBottom: '1.5rem'
}}>
  <div className="formula-box">
    $$
    Cost = \frac{a \cdot b}{c} \cdot \left( e^{c \cdot S_1} - e^{c \cdot S_0} \right)
    $$
  </div>
  <div className="formula-explanation">
    <div style={{ marginTop: '0' }}>
      <ul style={{ marginTop: '0.5rem', marginBottom: 0, paddingLeft: '1.5rem' }}>
        <li style={{ marginBottom: '0.5rem' }}>
          <code style={{ backgroundColor: 'transparent', padding: 0 }}>S₀</code> - circulating supply before the purchase (currentSupply)
        </li>
        <li style={{ marginBottom: '0.5rem' }}>
          <code style={{ backgroundColor: 'transparent', padding: 0 }}>S₁</code> - circulating supply after the purchase (S₀ + tokensToBuy)
        </li>
        <li style={{ marginBottom: '0.5rem' }}>
          Since b = c by design, the coefficient (a · b / c) simplifies to <code style={{ backgroundColor: 'transparent', padding: 0 }}>a</code>
        </li>
      </ul>
    </div>
  </div>
  <div style={{ marginTop: '1rem', paddingTop: '0.75rem', borderTop: '1px solid #e5e7eb', fontSize: '0.75rem', color: '#6b7280' }}>
    <a href="https://github.com/code-payments/flipcash-program/blob/main/api/src/curve.rs#L36-L54" style={{ color: '#502792' }}>Code Payments › flipcash-program › curve.rs:36-54</a>
  </div>
</div>

<div style={{ textAlign: 'justify' }}>

<p style={{ marginBottom: '1.5rem' }}>
The computed cost always exceeds (spot price × quantity) because each successive token in the transaction is priced marginally higher than the previous. This price impact is an inherent property of bonding curves and scales with transaction size relative to total supply.
</p>

</div>

### Tokens Bought for Value

<div style={{ textAlign: 'justify' }}>

<p style={{ marginBottom: '1.5rem' }}>
When participants specify a USDF amount to spend (the typical case), this inverse function determines the exact quantity of MNY received:
</p>

</div>

<div className="principle-card formula-card" style={{
  padding: '1.5rem',
  marginBottom: '1.5rem'
}}>
  <div className="formula-box">
    $$
    T = \frac{\ln\left(\frac{V}{a} + e^{c \cdot S_0}\right)}{c} - S_0
    $$
  </div>
  <div className="formula-explanation">
    <div style={{ marginTop: '0' }}>
      <ul style={{ marginTop: '0.5rem', marginBottom: 0, paddingLeft: '1.5rem' }}>
        <li style={{ marginBottom: '0.5rem' }}>
          <code style={{ backgroundColor: 'transparent', padding: 0 }}>T</code> - tokens received (tokensBought)
        </li>
        <li style={{ marginBottom: '0.5rem' }}>
          <code style={{ backgroundColor: 'transparent', padding: 0 }}>V</code> - USDF amount the participant wishes to spend
        </li>
        <li style={{ marginBottom: '0.5rem' }}>
          <code style={{ backgroundColor: 'transparent', padding: 0 }}>S₀</code> - circulating supply before the purchase
        </li>
        <li style={{ marginBottom: '0.5rem' }}>
          <code style={{ backgroundColor: 'transparent', padding: 0 }}>ln</code> - natural logarithm, the inverse of exponentiation (e^x)
        </li>
      </ul>
    </div>
  </div>
  <div style={{ marginTop: '1rem', paddingTop: '0.75rem', borderTop: '1px solid #e5e7eb', fontSize: '0.75rem', color: '#6b7280' }}>
    <a href="https://github.com/code-payments/flipcash-program/blob/main/api/src/curve.rs#L62-L74" style={{ color: '#502792' }}>Code Payments › flipcash-program › curve.rs:62-74</a>
  </div>
</div>

<div style={{ textAlign: 'justify' }}>

<p style={{ marginBottom: '1.5rem' }}>
The logarithmic term arises from inverting the exponential cost function. Because the natural log is the inverse of e^x, this formula algebraically solves for the supply endpoint that yields exactly the specified cost when integrated from S₀.
</p>

</div>

### Value from Selling Tokens

<div style={{ textAlign: 'justify' }}>

<p style={{ marginBottom: '1.5rem' }}>
The same logic applies in reverse when redeeming tokens. Tokens return to the reserve, supply decreases, and USDF is released:
</p>

</div>

<div className="principle-card formula-card" style={{
  padding: '1.5rem',
  marginBottom: '1.5rem'
}}>
  <div className="formula-box">
    $$
    Value = \frac{a \cdot b}{c} \cdot \left( e^{c \cdot S_0} - e^{c \cdot S_1} \right)
    $$
  </div>
  <div className="formula-explanation">
    <div style={{ marginTop: '0' }}>
      <ul style={{ marginTop: '0.5rem', marginBottom: 0, paddingLeft: '1.5rem' }}>
        <li style={{ marginBottom: '0.5rem' }}>
          <code style={{ backgroundColor: 'transparent', padding: 0 }}>S₀</code> - circulating supply before the redemption
        </li>
        <li style={{ marginBottom: '0.5rem' }}>
          <code style={{ backgroundColor: 'transparent', padding: 0 }}>S₁</code> - circulating supply after the redemption (S₀ - tokensToSell)
        </li>
        <li style={{ marginBottom: '0.5rem' }}>
          Returns the gross redemption value before protocol fees are applied
        </li>
      </ul>
    </div>
  </div>
  <div style={{ marginTop: '1rem', paddingTop: '0.75rem', borderTop: '1px solid #e5e7eb', fontSize: '0.75rem', color: '#6b7280' }}>
    <a href="https://github.com/code-payments/flipcash-program/blob/main/api/src/curve.rs#L36-L54" style={{ color: '#502792' }}>Code Payments › flipcash-program › curve.rs:36-54</a>
  </div>
</div>

<div style={{ textAlign: 'justify' }}>

<p style={{ marginBottom: '1.5rem' }}>
The mathematical symmetry between acquisition and redemption ensures that the reserve always holds sufficient USDF to honor all outstanding redemptions at current curve prices. This is a fundamental solvency invariant - the value locked in the reserve exactly equals the redemption value of all circulating tokens.
</p>

</div>

### Tokens Required for Value

<div style={{ textAlign: 'justify' }}>

<p style={{ marginBottom: '1.5rem' }}>
When participants specify a desired USDF amount to receive, this inverse function computes the MNY required to redeem. It operates on reserve value rather than supply:
</p>

</div>

<div className="principle-card formula-card" style={{
  padding: '1.5rem',
  marginBottom: '1.5rem'
}}>
  <div className="formula-box">
    $$
    T = \frac{1}{c} \cdot \left[ \ln\left(1 + \frac{V_0}{a}\right) - \ln\left(1 + \frac{V_1}{a}\right) \right]
    $$
  </div>
  <div className="formula-explanation">
    <div style={{ marginTop: '0' }}>
      <ul style={{ marginTop: '0.5rem', marginBottom: 0, paddingLeft: '1.5rem' }}>
        <li style={{ marginBottom: '0.5rem' }}>
          <code style={{ backgroundColor: 'transparent', padding: 0 }}>T</code> - tokens required to sell (tokensToSell)
        </li>
        <li style={{ marginBottom: '0.5rem' }}>
          <code style={{ backgroundColor: 'transparent', padding: 0 }}>V₀</code> - total USDF currently locked in the reserve
        </li>
        <li style={{ marginBottom: '0.5rem' }}>
          <code style={{ backgroundColor: 'transparent', padding: 0 }}>V₁</code> - reserve balance after redemption (V₀ - valueToReceive)
        </li>
        <li style={{ marginBottom: '0.5rem' }}>
          <code style={{ backgroundColor: 'transparent', padding: 0 }}>ln</code> - natural logarithm, the inverse of exponentiation (e^x)
        </li>
      </ul>
    </div>
  </div>
  <div style={{ marginTop: '1rem', paddingTop: '0.75rem', borderTop: '1px solid #e5e7eb', fontSize: '0.75rem', color: '#6b7280' }}>
    <a href="https://github.com/code-payments/flipcash-program/blob/main/api/src/curve.rs#L62-L74" style={{ color: '#502792' }}>Code Payments › flipcash-program › curve.rs:62-74</a>
  </div>
</div>

<div style={{ textAlign: 'justify' }}>

<p style={{ marginBottom: '1.5rem' }}>
This formulation uses the reserve's locked value as the state variable rather than circulating supply. The two are mathematically equivalent due to the one-to-one correspondence between supply levels and cumulative reserve value.
</p>

</div>

<Info>
Redemptions are subject to a fee (in basis points) deducted from the gross value. See the <a href="/protocol/proof-of-capital#fee-model" className="brand-link">Fee Model</a> section for details.

## Discrete Implementation

<div style={{ textAlign: 'justify' }}>

<p style={{ marginBottom: '1.5rem' }}>
The formulas above are mathematically exact, but how does the smart contract actually compute them? Direct calculation of exponentials and logarithms onchain is computationally expensive. Instead, the Moony Reserve precomputes all curve values at regular supply intervals and stores them in lookup tables. Transaction pricing reduces to table lookups with linear interpolation - efficient and producing identical results every time.
</p>

</div>
